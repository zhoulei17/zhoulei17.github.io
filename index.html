我们知道在搜索时，会调用IndexSearcher的其中一个search方法。我们的Solr大部分情况也会调用这个search方法。

public void search(Query query, Collector results)
  throws IOException {
  query = rewrite(query);
  search(leafContexts, createWeight(query, results.needsScores(), 1), results);
}

【Solr】-【SolrCloud】SolrCloud核心概念

发表于 2020-04-25 更新于 2020-04-28 分类于 Solr 阅读次数： Valine： 本文字数： 4.2k 阅读时长 ≈ 4 分钟

架构特性

SolrCloud架构主要有5个主要特性：

  • 可扩展性
  • 高可用性
  • 一致性
  • 简单性
  • 灵活性

可扩展性

扩展方法

扩容增加工作负载的处理能力，有两种方法可扩展：横向扩展、纵向扩展。

  • 纵向扩展：如，增加内容、升级CPU、使用固态加速I/O性能。

  • 横向扩展：分片、复制。

    • 分片：主要是为了加速索引、加速索引优化、增大单shard索引容量、并行搜索、方便索引备份。等诸多好处。

      • 你分片可以把大型索引分割成多个小索引，放到不同的shard中。

      • 允许用户同时操作多个shard索引，加快了索引和优化。

      • 多shard并行查，查完后再合并文档，一定程度上加快了搜索速度。

      • 多个shard分割以后，同时备份也方便、更快。

    • 复制：主要是保证shard对应节点的可靠性、分担单节点复杂搜索压力，加快搜索。

      • 假如一个shard由10个副本组成，客户端同时有10个查询请求访问集群，（理想情况）每个副本都可以处理这个shard的查询。处理能力增加10倍。

      • 多个副本起到了容灾容错的作用，一个副本挂了，其他副本还可以服务。

    一般横向扩展更受欢迎，因为更经济（比如亚马逊的EC2）。可以用多台配置没有那么高的机器并行处理请求充分榨干单机性能，组合达到的性能一般比同等能力的单机便宜。

    但是一味的通过横向加机器，等业务发展到一定程度也不一定能解决问题，因为一个是机器越来越多，管理起来也越来越困难，另外一个是业务复杂以后，本身单机性能也要够用（因为不能无限制扩机器且单请求也可能很复杂）。在复杂性、成本等因素下，横向与纵向的扩展方法需要综合考虑。

同时为了到达，最优性能，撇开代码级别与业务级别的优化不讲，也需要有足够的内容将索引缓存在操作系统缓存中，还需要有足够内容来容纳JVM缓存、排序、分面。如果索引中可以搜索的数据需要有20GB，那也还需要8GB内存给JVM，总共要28GB内容。Solr很明显也是I/O密集型的应用（特别是索引文档时），需要性能更好的固态硬盘。

【Solr】-【SolrCloud】分布式索引

发表于 2020-04-25 更新于 2020-04-26 分类于 Solr 阅读次数： Valine： 本文字数： 4.3k 阅读时长 ≈ 4 分钟

SloudCloud中shard中的索引与单机中的索引文档是一样的结构，且一个shard的多个replicas理想状态下是完全一样的。从客户端角度看，在访问单机索引或分布式索引客户端是不感知的。从Solr服务器角度看，原来单机的索引被分割为多个shard索引目录，就单个shard来说，都是一个完整的索引结果，都可以单独访问。

SolrCloud中分布式索引的总体目标是能够把文档发送到集群的的任何节点上，并且能把文档在正确的分片中被索引。此外，SolrCloud的分布式索引目的在于，能够避免单点故障，达到高可用。

文档的分片

当Solr索引新文档时，需要将其分配到一个分片上。Solr使用文档管理器组件来确认文档应当被分到哪个分片上。SolrCloud支持两种文档路由策略：

  • compositeID(默认)
  • 隐式路由。用户可以自定义路由策略。

查看/clusterstate.json，可以看到笔者所在公司collection由16个shard组成。每个分片会分配一个32位的散列值，如上图，shar1的散列区间是80000000-8fffffff，而shard2的散列区间是90000000-9fffffff(十六进制数)。系统在每个分片之间均分这32位散列值区间。

默认的compositeID路由器会计算出文档里唯一ID字段的数值散列值，并将文档分配给哈希值区间包含计算得到的哈希值的分片。每个发送给SolrCloud的文档都必须具有唯一的文档ID号。

【Lucene】-【Search】Collector（四）TopDocsCollector及其子类TopFieldCollector

发表于 2020-04-13 更新于 2020-04-19 分类于 Lucene 阅读次数： Valine： 本文字数： 5.8k 阅读时长 ≈ 5 分钟

在上一篇Collector(二)的文章中，介绍了TopDocsCollector的其中一个子类TopScoreDocCollector收集器，以及TopScoreDocCollector的两个子类SimpleTopScoreDocCollector、PagingTopScoreDocCollector，它们的排序规则是“先打分、后文档号”。

这次，介绍TopDocsCollector的另外一个子类TopFieldCollector，它的排序规则是“先域比较（FieldComparator），后文档号”。

  • 先域比较（FieldComparator）：根据文档（Document）中的排序域（SortField）的阈值进行排序。
  • 后文档号：由于文档号是唯一的，当无法通过域比较来获得顺序关系时，可以再通过文档号进行排序，文档号越小，排名越靠前。

下面通过一个例子先介绍怎么使用TopFieldCollector，然后再来介绍排序的原理。

在搜索阶段如果使用了域排序，那么Lucene默认使用的是TopFieldCollector。
